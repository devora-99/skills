# Project AGENTS

## gstack

이 프로젝트에서는 사용자 프롬프트의 의도를 해석한 뒤, 아래 유형에 해당하면 일반 처리보다 `gstack` 스킬 사용을 반드시 우선한다.

### 강제 규칙

- 사용자가 특정 `gstack-*` 스킬명을 직접 언급하면 해당 스킬을 반드시 사용한다.
- 요청이 아래 매핑 규칙과 명확히 일치하면 일반 방식으로 처리하지 말고 해당 `gstack` 스킬을 반드시 먼저 사용한다.
- 웹 브라우저 조작, QA, 보안 점검, 코드 리뷰, 원인 분석은 기본적으로 `gstack` 담당 영역으로 간주한다.
- 여러 스킬 후보가 겹치면 가장 직접적으로 문제를 해결하는 스킬 1개를 먼저 사용하고, 필요할 때만 다음 스킬로 이어간다.
- 스킬이 실제로 없거나 실행 불가능한 경우에만 일반 방식으로 대체하며, 그때는 이유를 짧게 알린다.

### 상황별 강제 매핑

- 웹페이지 열기, 클릭, 입력, 스크린샷, 브라우저 재현, 콘솔 확인, 실제 사이트 점검 요청: `gstack-browse` 반드시 사용
- staging/production URL 테스트, 사용자 플로우 점검, 회귀 테스트, 버그 재현과 수정 요청: `gstack-qa` 반드시 사용
- 코드 변경 없이 QA 리포트만 필요한 요청: `gstack-qa-only` 반드시 사용
- 코드 리뷰, 배포 전 위험 점검, 회귀 가능성 확인, 숨은 버그 탐지 요청: `gstack-review` 반드시 사용
- 버그 원인 분석, 재현 경로 추적, 근본 원인 조사 요청: `gstack-investigate` 반드시 사용
- 보안 취약점 점검, 공격 시나리오 검토, OWASP/STRIDE 관점 감사 요청: `gstack-cso` 반드시 사용
- 기능 설계 검토, 아키텍처 점검, 데이터 흐름 검토, 테스트 전략 검토 요청: `gstack-plan-eng-review` 반드시 사용
- 제품 방향, 기능 범위, 우선순위, founder/CEO 시각 검토 요청: `gstack-plan-ceo-review` 반드시 사용
- UI/UX 평가, 디자인 품질 점검, AI 슬롭 감지 요청: `gstack-plan-design-review` 반드시 사용
- 디자인 수정까지 포함한 종합 디자인 리뷰 요청: `gstack-design-review` 반드시 사용
- 배포 준비, PR 정리, 릴리즈 전 검증 요청: `gstack-ship` 반드시 사용
- 배포 실행 후 상태 확인까지 포함된 요청: `gstack-land-and-deploy` 반드시 사용
- 배포 환경 설정이나 배포 흐름 초기 구성 요청: `gstack-setup-deploy` 반드시 사용
- 브라우저 로그인 세션, 쿠키 설정, 인증 상태 준비 요청: `gstack-setup-browser-cookies` 반드시 사용
- 성능 측정, 웹 성능 비교, 로드 시간 분석, 리소스 분석 요청: `gstack-benchmark` 반드시 사용
- 배포 후 모니터링과 이상 징후 확인 요청: `gstack-canary` 반드시 사용
- 문서 최신화, 릴리즈 문서 정리 요청: `gstack-document-release` 반드시 사용

### 우선순위

1. 사용자가 직접 지정한 `gstack-*` 스킬
2. 프롬프트 의미와 일치하는 강제 매핑 스킬
3. 그 외 일반 처리

### 운영 원칙

- 사용자가 별도 명령어를 외우지 않아도 되도록, 프롬프트 의미만 분명하면 해당 `gstack` 스킬을 자동 선택한다.
- 일반 응답으로 처리 가능한 요청이어도 위 강제 매핑에 해당하면 먼저 `gstack` 스킬을 따른다.
- 중요한 작업에서 사용자가 스킬명을 직접 적으면 그 지시를 최우선으로 따른다.
